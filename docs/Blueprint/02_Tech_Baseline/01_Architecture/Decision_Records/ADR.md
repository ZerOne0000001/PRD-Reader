# 🧠 架构决策记录 (ADR: Architecture Decision Records)

本目录记录了系统在演进过程中做出的关键技术决策，帮助 AI 了解“为什么这么做”而不是“做了什么”，避免在后续迭代中提出已被否决的设计方案。

---

## ADR-001: 本地 JSON 存储替代数据库

### 背景
系统需要存储极少量的配置信息：平台实例 URL、全局 Token 和十几个白名单仓库信息。

### 决策
放弃引入 MySQL / SQLite / Redis 等任何形式的数据库，直接使用 Node.js 后端根目录下的 `config.json` 文件进行读写持久化。

### 理由
1.  **极简主义**：本项目定位是轻量级内网工具，零外部依赖（除了 Node.js）是最高优先级，做到“拉下代码 npm run dev 即用”。
2.  **配置低频**：白名单和 Token 配置修改极不频繁，JSON 文件的并发写入冲突概率几乎为零。

### 约束
在 `configService.ts` 中，所有的写操作使用同步的 `fs.writeFileSync`，确保在极低并发下的一致性。

## ADR-002: 虚拟静态资源路由 (Virtual Static Server)

### 背景
当用户在左侧文件树点击 `.html` 原型文件或图片时，如果按常规逻辑拉取 Raw 数据（字符串/Buffer）存入前端的 Zustand State，会导致大型 HTML 瞬间撑爆浏览器内存，且内部相对路径资源引用会解析失败（404）。

### 决策
在 Express 后端设计专门的 `/api/repo/raw/:projectId/*` 通配符路由，模拟真实的静态服务器。前端状态机不再拉取内容，而是直接拼接生成该路由的 URL，并赋值给 `iframe` 的 `src` 或 `img` 的 `src`。

### 理由
1.  **利用浏览器原生机制**：让浏览器自己去处理资源的流式加载和缓存，不经过前端业务代码的内存。
2.  **相对路径兼容**：`iframe` 的 `src` 加载时，其内部相对请求会自动补全，恰好被后端的通配符路由接管并正确代理到代码托管平台。

### 约束
必须严格根据文件扩展名设置正确的 `Content-Type` 头，否则浏览器可能拒绝渲染 HTML 或图片。

## ADR-003: 强制分离前后端职责（Token 绝对安全）

### 背景
既然是 Web 平台，前端直接调用平台 API 是最省事的。

### 决策
严禁前端发起任何直接指向代码托管平台（GitLab/GitHub）的跨域请求。必须建立 Node.js Express 代理层。

### 理由
为了实现“开箱即用”和“免登录分享”，系统必须使用全局只读 Token。如果前端直连，Token 必然暴露在浏览器的 Network 面板和 JS 代码中，引发严重的安全事故。通过 Node.js 代理，Token 永远驻留在服务器，前端一无所知。
