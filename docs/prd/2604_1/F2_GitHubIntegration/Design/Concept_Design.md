# 概念设计 (Concept Design)

## 1. 设计方向概述

本次迭代的核心目标是**在现有 GitLab 文件查看系统的基础上，增加 GitHub 平台支持**，实现平台灵活切换。

### 1.1 关键设计决策

| 决策项 | 选定方案 | 理由 |
|:---|:---|:---|
| **平台选择器交互** | Tab 页签形式 | 两个平台配置一目了然，切换直观 |
| **白名单管理** | 动态过滤展示 | 列表简洁，根据当前平台显示对应仓库 |
| **路由抽象** | 通用路由 `/api/repo/*` | 统一接口，便于扩展更多平台 |
| **服务层** | 独立 `githubService.ts` | 职责单一，便于维护和扩展 |

│  │ Token:    [•••••••••••••••           ] [👁]         │    │
│  └─────────────────────────────────────────────────────┘    │
│                          [测试连接]  [保存配置]               │
│                                                              │
│  仓库白名单 (GitLab)                                         │
│  ┌─────────────────────────────────────────────────────┐    │
│  │ [+] 添加仓库: [________________________] [添加]      │    │
│  │ ┌───────────────────────────────────────────────┐   │    │
│  │ │ 📦 repo-a           ✓正常            🗑️      │   │    │

**优点**：
- 两个平台的配置同时可见，便于对比和检查

**缺点**：
- 占用页面纵向空间较大
- 需要维护两套配置表单的 UI 代码

### 2.2 白名单管理：动态过滤展示

```
当前平台: GitLab ▼  ← 全局平台切换器

仓库白名单 (GitLab)
┌─────────────────────────────────────────────────────────┐
│ 添加仓库: [owner/repo________________] [添加]            │
│ ┌───────────────────────────────────────────────────┐  │
│ │ 📦 PRD-Document     ✓正常              🗑️        │  │
│ │ 📦 Design-Assets    ✓正常              🗑️        │  │
│ │ 📦 API-Docs         ✗异常              🗑️        │  │
│ └───────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────┘
```

**优点**：
- 列表简洁清晰，不会因为跨平台仓库混杂而混乱
- 每个平台维护独立的仓库列表，职责清晰
- 用户能快速识别当前平台的仓库状态

**缺点**：
- 需要切换平台才能看到另一平台的仓库列表
- 如果用户需要同时管理两个平台的仓库，操作步骤会增加

## 3. 概念方案总结

### 3.1 选定的概念方向

本次设计采用 **Tab 页签 + 动态过滤** 的组合方案：

1. **平台选择**：使用 Tab 页签展示 GitLab 和 GitHub 两个配置区块
2. **白名单管理**：根据当前选中的平台 Tab，动态显示对应平台的仓库白名单
3. **配置持久化**：两个平台的配置独立存储，通过 `platform` 字段标识当前使用的平台

### 3.2 设计原则

| 原则 | 说明 |
|:---|:---|
| **渐进式披露** | 用户切换 Tab 时，下方表单和白名单自动切换，减少视觉干扰 |
| **状态清晰** | 当前平台状态始终可见，错误状态明确提示 |
| **操作闭环** | 配置保存后自动刷新白名单，无需手动刷新页面 |
| **容错优先** | Token 失效时文件树区域显示友好提示，引导管理员检查配置 |

### 3.3 关键交互流程

1. **首次配置 GitHub**：
   - 点击"GitHub" Tab → 填写 GitHub 实例地址（默认 github.com）和 Token → 测试连接 → 保存配置 → 添加仓库白名单

2. **切换平台**：
   - 点击"GitHub" Tab → 系统自动加载 GitHub 配置 → 显示 GitHub 仓库白名单 → 前台自动刷新显示 GitHub 仓库

3. **处理异常**：
   - 文件树加载失败 → 显示错误提示"连接失败，请联系管理员检查配置" → 管理员进入后台 → 查看 Token 状态 → 更新配置

## 4. 技术架构概念

### 4.1 数据流

```
┌─────────────┐    ┌─────────────┐    ┌─────────────────┐
│  Admin.tsx  │───▶│ configStore │───▶│  /api/config    │
└─────────────┘    └─────────────┘    └─────────────────┘
                          │                     │
                          ▼                     ▼
                   ┌─────────────┐    ┌─────────────────┐
                   │ platform    │    │  config.json    │
                   │ 字段决定    │    │  (双平台配置)   │
                   │ 调用服务    │    └─────────────────┘
                          │
          ┌───────────────┼───────────────┐
          ▼               ▼               ▼
   ┌─────────────┐ ┌─────────────┐ ┌─────────────┐
   │ gitlab      │ │ github      │ │ repo Router │
   │ Service     │ │ Service     │ │ (统一分发)  │
   └─────────────┘ └─────────────┘ └─────────────┘
```

### 4.2 路由抽象

| 前端请求 | 后端路由 | 实际服务 |
|:---|:---|:---|
| 获取文件树 | GET /api/repo/tree | 根据 platform 调用 GitLab/GitHub Service |
| 获取文件内容 | GET /api/repo/file | 根据 platform 调用对应 Service |
| 获取静态资源 | GET /api/repo/raw/* | 根据 platform 调用对应 Service |

---

## 5. 结论

本概念设计明确了以下核心方向：

1. **交互设计**：采用 Tab 页签 + 动态过滤的组合方案，兼顾直观性和简洁性
2. **技术架构**：抽象通用路由，独立服务文件，便于扩展和维护
3. **用户体验**：平台切换无缝衔接，错误提示清晰友好

此设计在满足当前需求的同时，为未来接入更多平台（GitLab Enterprise 等）预留了扩展空间。
