# 需求背景 (Requirement Background) - F2 文档查询功能

## 1. 业务目标

### 1.1 核心目标
为 PRD-Reader 系统增加**跨仓库文档检索能力**，让用户能够通过关键词快速定位目标文档，而非依赖低效的文件树逐层展开浏览。

### 1.2 成功指标
- 用户查找文档的平均时间缩短 50% 以上
- 搜索功能使用率达到活跃用户的 30% 以上

## 2. 背景与痛点

### 2.1 现状问题
PRD-Reader 当前仅支持通过左侧抽屉式文件树逐层展开浏览文档。当系统配置了多个仓库、或单个仓库目录层级较深时，用户需要反复点击展开才能找到目标文件，效率极低。

### 2.2 用户痛点
- **定位困难**：不知道目标文档在哪个仓库、哪个目录下
- **浏览低效**：文件树层级深，需要多次点击展开
- **无法检索**：无法通过文档内容关键词反向定位文件

## 3. 用户画像 (User Personas)

| 角色 | 特征 | 核心痛点 | 行为限制/偏好 |
| :--- | :--- | :--- | :--- |
| **产品经理 (PM)** | 非技术背景，日常需要查阅大量 PRD、设计文档、会议纪要等 Markdown 文档 | 记不清文档具体位置，只记得大概内容或关键词 | 偏好简洁直观的界面，对复杂操作有抵触 |
| **技术人员** | 熟悉 GitLab 操作，需要查阅技术文档、API 文档、架构设计等 | 需要快速定位特定技术关键词所在的文档 | 习惯使用搜索功能，对响应速度有要求 |

## 4. 核心场景 (Core Scenarios)

| 场景名称 | 关联用户画像 | 触发条件与环境 | 当前痛点 | 核心动作 | 期望结果 | 潜在异常 |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| **跨仓库文档检索** | PM、技术人员 | 用户记得文档中的某个关键词，但不确定在哪个仓库 | 需要逐个仓库、逐层展开查找 | 输入关键词 → 点击搜索 → 查看结果列表 → 点击打开 | 快速定位到目标文档 | 搜索结果过多，难以筛选 |
| **文件名快速定位** | PM、技术人员 | 用户知道文件名的一部分，想快速找到该文件 | 文件树层级深，展开耗时 | 输入文件名关键词 → 点击搜索 → 查看匹配的文件列表 | 直接跳转到目标文件 | 文件名相似度过高，结果不精确 |
| **内容关键词追溯** | PM | 用户想找到提到某个功能/需求的文档 | 不知道哪个文档提到过 | 输入功能名称 → 点击搜索 → 查看内容片段预览 | 通过预览判断是否为目标文档 | 关键词在多个文档中出现，需要逐个查看 |

## 5. 功能边界

### 5.1 包含范围
- 搜索入口：复用现有顶部导航栏搜索框
- 搜索范围：所有授权仓库中的 `.md` 文件内容 + 所有文件的文件名
- 搜索结果：独立搜索结果页面，支持分页、内容片段预览、关键词高亮
- 结果交互：点击结果在新标签页打开 Reader 页面并定位到对应文件

### 5.2 不包含范围
- 实时搜索（边输入边搜索）
- 搜索历史记录
- URL 分享功能（搜索结果页面不支持通过 URL 复用）
- 非 `.md` 文件的内容检索（如 `.html`、`.txt` 等）
