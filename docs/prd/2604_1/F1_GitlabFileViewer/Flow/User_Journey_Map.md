# 用户旅程地图 (User Journey Map)

## 核心任务：产品经理通过平台查找并阅读某次迭代的 PRD (Markdown) 和 高保真原型 (HTML)

| 阶段 | 触点 | 用户行为 (用户想要做什么？) | 系统行为 (系统如何响应？) | 当前痛点与机会点 |
| :--- | :--- | :--- | :--- | :--- |
| **1. 发现与进入** | 平台首页 / 浏览器书签 | 1. 产品经理打开浏览器，输入平台地址或点击书签。<br>2. 系统可能处于未登录状态或已静默使用全局只读账号。 | 1. 系统加载首页，直接展示已配置的“可见仓库”列表。<br>2. 后台通过配置好的全局只读账号连接 GitLab API。 | **痛点**: 原生 GitLab 需要每次个人登录并寻找项目。<br>**机会点**: 直接通过统一入口和全局账号静默加载，消除个人鉴权障碍，做到“开箱即用”。 |
| **2. 搜索与定位** | 首页搜索框 / 左侧文件树 | 1. PM 记得需求代号，在顶部全局搜索框输入“2604_1”。<br>2. 或者，在左侧图形化文件树中展开“需求文档仓库” -> 找到对应迭代文件夹。 | 1. 系统调用 GitLab API 进行跨仓库的全文搜索，并实时高亮展示匹配结果列表。<br>2. 文件树以异步懒加载方式请求 GitLab 目录结构，左侧展示带有图标（如文件夹、MD、HTML 图标）的图形化层级。 | **痛点**: 跨项目找文件如大海捞针。<br>**机会点**: 提供极速的跨项目全文搜索；文件树交互应像本地资源管理器一样流畅。 |
| **3. 阅读与渲染 (Markdown)** | 搜索结果列表 / 文件树节点 | 1. PM 点击搜索结果或文件树中的 `Requirement_Background.md`。 | 1. 系统高亮选中左侧树节点。<br>2. 右侧主区域显示“加载中”骨架屏。<br>3. 系统获取 Markdown 源码，使用解析器渲染为排版精美的网页，并识别 ````mermaid` 代码块进行图形化渲染。 | **痛点**: 原生界面 Markdown 样式单一，Mermaid 支持可能存在版本兼容问题。<br>**机会点**: 提供更沉浸式的阅读主题（如暗黑/明亮模式），确保所有 Mermaid 图表都能完美渲染。 |
| **4. 演示与预览 (HTML)** | 文件树节点 | 1. PM 需要给技术演示高保真交互，在同目录下点击 `Prototype.html`。 | 1. 右侧主区域切换内容，不再渲染源码，而是通过 iframe 隔离渲染该单文件 HTML 的网页视图。<br>2. 提供一个“在新窗口全屏打开”的快捷按钮。 | **痛点**: 必须下载 HTML 文件到本地才能用浏览器打开查看效果。<br>**机会点**: 做到所见即所得，单文件 HTML 直接在平台内嵌预览，大幅提升沟通演示效率。 |
| **5. 结束与分享** | 浏览器地址栏 / 页面右上角按钮 | 1. PM 确认文档和原型无误后，想要将这个页面分享给开发人员。 | 1. 系统的 URL 必须随当前打开的文件路径实时变化（如 `?repo=xxx&file=path/to/file.md`）。<br>2. PM 复制当前 URL 并发送给技术。技术打开该链接，直接定位到该文件。 | **痛点**: 分享路径复杂，对方可能没有该仓库权限。<br>**机会点**: 因为是全局只读账号，分享链接必定可以被团队内任何人无缝打开查看。确保 URL 的深层链接 (Deep Linking) 能够准确还原当前文件树状态和右侧内容。 |
