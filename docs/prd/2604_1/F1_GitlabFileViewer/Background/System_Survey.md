# 系统调研 (System Survey)

## 1. 系统定位与核心逻辑
*   **系统类型**: 全新独立系统 (基于 GitLab 的 Web 端文件查看平台)。
*   **核心逻辑**: 
    *   **鉴权与访问**: 系统后端配置一个具有特定仓库只读权限的 GitLab 账号 (Token)。前端用户访问该系统时，无需个人登录（或仅需极简单的内部系统单点登录，具体视部署环境而定），系统代理请求 GitLab API 获取文件数据。
    *   **数据流向**: 用户请求 (搜索/浏览) -> 后端代理 (带全局 Token) -> GitLab API -> 返回数据 (树结构/文件内容) -> 前端渲染展示。
    *   **渲染机制**: 
        *   **Markdown**: 前端获取 raw 内容，使用解析库 (如 `marked.js`) 转换为 HTML，并集成 `mermaid.js` 拦截和渲染 ````mermaid` 代码块。
        *   **HTML**: 单文件 HTML 直接在前端通过 `iframe` 隔离渲染，确保其内联样式和脚本正常运行，不影响主系统。

## 2. 约束与依赖
*   **强依赖**: 极度依赖 GitLab API (V4) 的稳定性和响应速度（特别是跨项目搜索 `GET /search` 和获取文件树 `GET /projects/:id/repository/tree`）。
*   **性能瓶颈预警**: GitLab 的跨项目全局搜索 API 可能较慢，前端需要良好的 Loading 状态提示；对于超大 Markdown 文件或包含海量节点的 HTML 文件，前端渲染可能存在性能问题。
*   **安全约束**: 虽然是全局只读账号，但必须在系统层面（后端或配置层）严格限制**只能访问指定的白名单仓库**，绝对不能将 GitLab Token 暴露给前端，所有请求必须通过后端代理转发。

## 3. 受影响的现有页面/系统
*   **无**: 作为一个全新的独立查阅平台，不影响现有任何系统的正常运行。
