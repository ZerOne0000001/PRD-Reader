# 需求背景 (Requirement Background)

## 1. 业务目标与背景
当前团队中，产品经理和技术人员需要频繁查看存放在 GitLab 中的 PRD 文档（Markdown 格式）以及设计原型/静态页面（HTML 格式）。然而，GitLab 原生的界面体验更偏向于代码托管与开发管理，对于非纯代码文档的阅读体验欠佳，且不支持 HTML 文件的直接预览渲染。
因此，需要开发一套“基于 GitLab 的文件查看系统”，旨在通过配置一个全局只读账号，为产品和技术团队提供一个交互更友好、更偏图形化的 Web 阅读平台。该平台专注于跨项目的文件检索与文档沉淀查看，屏蔽复杂的 Git 操作，提升团队内的文档查阅效率。

## 2. 用户画像 (User Personas)

| 角色 | 核心特征 | 核心痛点 | 行为偏好与限制 |
| :--- | :--- | :--- | :--- |
| 产品经理 (PM) | 频繁查阅 PRD、流程图及交互原型。不熟悉 Git 命令行，依赖图形化界面。 | 1. GitLab 上 Markdown 渲染效果一般，无法直接预览 HTML 原型；<br>2. 跨仓库找文档困难。 | 偏好直观的文件夹树状结构和所见即所得的文档渲染体验；希望有全局搜索能快速定位文档。 |
| 技术人员 (Dev) | 查阅需求文档进行开发评估。熟悉 GitLab 但希望有更纯粹的阅读体验。 | 在开发过程中需要频繁对照 PRD 和原型，GitLab 原生界面跳转繁琐，且无法直观查看 Mermaid 架构图和 HTML 演示。 | 追求高效的跨项目全文检索，要求 Markdown 和 Mermaid 图表能准确渲染。 |

## 3. 核心场景 (Core Scenarios)

| 场景名称 | 关联用户画像 | 触发条件与环境 | 当前痛点 | 核心动作 | 期望结果 | 潜在异常 |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| 配置可见仓库范围 | 系统管理员 | 系统初始化或新增业务线时，在后台进行配置。 | 默认全量拉取可能包含敏感代码仓库，信息噪音大。 | 通过后台界面配置该平台允许读取的 GitLab 仓库列表。 | 前台仅展示配置白名单内的仓库及其文件树。 | 仓库不存在或只读账号无权限访问。 |
| 跨项目全文搜索 | PM, Dev | 记不清文档在哪个仓库，只记得某个关键词（如需求名）时。 | 只能逐个仓库点进去找，耗时费力。 | 在首页搜索框输入关键词进行全局搜索。 | 返回包含该关键词的文件列表，点击可直接跳转到文档详情。 | 搜索结果过多导致渲染卡顿，或索引延迟导致搜不到最新文件。 |
| 沉浸式阅读 Markdown (含 Mermaid) | PM, Dev | 评审或开发阶段，需要仔细阅读 PRD 及其中的流程图时。 | GitLab 原生 Markdown 对复杂 Mermaid 图表支持不佳，阅读体验差。 | 在文件树中点击 `.md` 文件。 | 页面右侧渲染出排版精美的文档内容，Mermaid 流程图/架构图清晰可见。 | Markdown 语法不规范导致渲染错乱。 |
| 预览 HTML 静态页面 | PM, Dev | 需要查看前端切图或高保真交互原型时。 | GitLab 只能查看 HTML 源码，无法预览网页效果。 | 在文件树中点击 `.html` 单文件。 | 在系统内（或新标签页）直接以网页形式渲染该 HTML 文件，展示原型效果。 | HTML 文件内部引用了相对路径的外部 CSS/JS 资源导致加载失败（需明确仅支持单文件 HTML 渲染）。 |
