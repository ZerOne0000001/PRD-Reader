# 概念设计 (Concept Design)

## 1. 设计目标与挑战
*   **目标**: 为非开发人员（特别是产品经理）提供一个更像本地文件夹的、极速且沉浸式的 Web 文档阅读体验。
*   **挑战**: 
    1.  如何将跨项目的零散文档聚合在一个直观的界面中。
    2.  如何让长篇 PRD（Markdown + Mermaid）的阅读体验最佳化。
    3.  如何优雅且安全地预览单文件 HTML。

## 2. 方案推演 (两种方向)

### 方案 A: 经典 IDE 风格 (左右分栏，侧边树 + 主视图) - 推荐
*   **交互模式**: 类似 VS Code 或 GitLab Web IDE 的布局。左侧固定为仓库/文件树结构，支持多层级展开；顶部为全局搜索框；右侧为主视图区（默认展示欢迎页或最近文档，点击左侧文件后右侧渲染内容）。
*   **优点**: 
    *   **符合心智模型**: 产品和技术人员对这种“左树右内容”的结构非常熟悉。
    *   **极简导航**: 在阅读一篇文档时，随时可以点击左侧查看同目录下的其他文档（如从 PRD 切换到 HTML 原型）。
    *   **空间利用率高**: 右侧留有大量空间用于沉浸式阅读，适合长篇 Markdown。
*   **缺点**: 在移动端屏幕下体验较差（但本需求主要面向 PC 端工作场景，可接受）。

### 方案 B: 知识库/维基风格 (卡片式首页 + 全屏阅读页)
*   **交互模式**: 首页展示配置的“可见仓库”卡片，点击卡片进入该仓库的列表页，再点击文件进入全屏沉浸式阅读页（带面包屑导航）。
*   **优点**: 首页视觉冲击力强，更像一个正式的知识管理系统。
*   **缺点**: 
    *   **层级太深**: 频繁在不同文件间切换时需要多次返回上一级，效率低下。
    *   **跨文件关联弱**: 无法像文件树那样直观看到同级目录下的所有相关文件。

## 3. 最终决策与选择原因
**选择方案 A (经典 IDE 风格，左树右内容)**。

**原因**:
*   本系统的核心诉求是**“高频的文件查看与切换”**。产品和技术在核对需求时，经常需要在 `.md` 和 `.html` 之间来回切换。方案 A 的左侧文件树能完美支持这种操作，无需跳转页面。
*   结合顶部全局搜索，可以做到“左侧找文件，顶部搜全文，右侧看内容”的无缝衔接。
*   右侧阅读区可以轻松实现 Markdown 和 HTML iframe 的状态切换。
