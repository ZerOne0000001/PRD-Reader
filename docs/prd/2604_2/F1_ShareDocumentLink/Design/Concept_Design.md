# 概念设计与方案推演 (Concept Design)

## 1. 需求回顾
* **核心目标**：实现系统内部分享链接功能，让接收者通过带参数的 URL 能够“开箱即用”地直达具体的仓库文件或文件夹，彻底消除寻找成本。
* **关键约束**：在现有 `Reader.tsx` 的单页路由架构下进行无缝集成；左侧树需要支持“下钻”加载模式；404 异常需在阅读区原位拦截。

## 2. 交互方案推演：分享入口设计

### 方案 A：左侧树节点原位替换 (Hover Icon -> Popover)
* **设计说明**：用户鼠标悬停在左侧文件树的某一行时，点击原有的“复制 Git 地址” Icon。点击后，Icon 附近弹出一个小型气泡菜单（Popover），提供“复制 Git 地址”和“复制系统分享链接”两个选项。
* **优点 (Pros)**：
  - 不增加新的常驻 UI 元素，保持界面的极简和干净。
  - 操作符合直觉，用户已经在树上定位到了文件，顺手就能分享。
* **缺点 (Cons)**：
  - 隐藏得比较深，新用户可能不知道悬停后还能调出二级菜单。

### 方案 B：右侧阅读区顶部常驻分享按钮 (Header Action)
* **设计说明**：在右侧主阅读区的顶部导航栏（面包屑旁边），新增一个常驻的“分享”按钮。点击后直接复制当前正在阅读的文档链接，或弹出选项菜单。
* **优点 (Pros)**：
  - 曝光率极高，功能入口显性化，用户一眼就能看到。
* **缺点 (Cons)**：
  - 只能分享**当前正在阅读的文件**，无法分享**文件夹**（因为点击文件夹不会在右侧渲染内容）。这与用户要求的“能精确到文件或者文件夹就行了”存在逻辑冲突。

### 决策结论：选择【方案 A】
* **原因**：为了满足“分享文件夹”的需求，分享操作的载体必须绑定在左侧的树节点上。方案 A 完美契合现有的交互逻辑（原有的复制 Git 地址功能已经在树节点上），只需要将直接复制改为弹出菜单即可，开发成本低且体验连贯。

## 3. 技术方案推演：URL 状态同步机制

### 方案 A：URL 驱动状态 (Single Source of Truth)
* **设计说明**：将 URL 参数（`?repo=xxx&file=xxx`）作为系统的**唯一事实来源**。点击树节点时，首先通过 `react-router` 改变 URL；然后由一个全局的 `useEffect` 监听到 URL 变化，再去触发状态机 `readerStore` 加载对应的数据。
* **优点 (Pros)**：
  - 架构最纯粹，完美支持浏览器的前进/后退操作。
* **缺点 (Cons)**：
  - 改造现有的 `readerStore.fetchFile` 逻辑成本较高，可能会引发不可预知的组件重渲染风暴。

### 方案 B：状态驱动 URL (State Syncing) - 推荐
* **设计说明**：保持现有 `readerStore` 的核心流转逻辑不变。当用户点击树节点时，依然是直接调用 `fetchFile` 加载数据；在数据加载成功或选中节点变化时，**静默调用** `history.replaceState` 或 `setSearchParams` 去更新地址栏 URL。
* **同时**，在应用首次加载时（`mount`），读取一次 URL 参数，如果存在，则主动触发状态机的“下钻加载”与“文件拉取”。
* **优点 (Pros)**：
  - 对现有代码的侵入性最小，稳妥可靠。
  - 初始化加载与日常点击浏览的逻辑解耦，便于单独处理“下钻”这种特殊情况。
* **缺点 (Cons)**：
  - 浏览器前进/后退的历史记录管理可能需要额外写一点粘合代码。

### 决策结论：选择【方案 B】
* **原因**：作为“现有系统的迭代/修改”，在保证功能交付的前提下，应当尽量选择对核心数据流侵入性最小的方案。方案 B 的静默同步机制足以满足“复制当前地址栏”的用户痛点，且改造风险极低。
