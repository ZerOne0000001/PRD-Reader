# 用户故事 (User Stories)

## Story 1: URL 实时响应当前阅读路径
**作为** 一个系统使用者
**我希望** 在点击左侧文件树的不同节点（文件或文件夹）时，浏览器的地址栏能实时更新
**以便于** 我可以直接复制浏览器地址栏的链接发送给他人。

**验收标准 (Acceptance Criteria):**
1. 点击任何有效的文件或文件夹节点，浏览器 URL 自动附加/更新查询参数（例如：`?repo=项目ID&file=文件相对路径`）。
2. URL 的更新必须是无刷新的（使用 History API，如 `react-router` 的 `useSearchParams`），不打断当前的阅读体验。
3. 如果点击的是仓库根节点，URL 仅包含 `repo` 参数。
4. 使用浏览器的“前进/后退”按钮时，系统能根据历史 URL 正确重新渲染对应的文件和树状高亮。

## Story 2: 统一的分享入口与选项弹窗
**作为** 一个分享者（如产品经理）
**我希望** 点击文件树节点上的分享 Icon 后，能弹出一个选项菜单
**以便于** 我可以选择复制“Git 原地址”还是“本系统的分享链接”。

**验收标准 (Acceptance Criteria):**
1. 鼠标悬停在左侧文件树节点时，显示原有的分享 Icon。
2. 点击该 Icon 后，不再直接复制 Git 地址，而是弹出一个小型菜单（Popover/Dropdown 或 Modal）。
3. 菜单内至少包含两个选项：“复制 Git 地址”、“复制系统分享链接”。
4. 点击“复制系统分享链接”后，系统读取当前状态拼装完整的带参数 URL 并写入剪贴板。
5. 复制成功后，提供明确的视觉反馈（如 Toast 提示“链接已复制”）。

## Story 3: 深度链接直达 (Deep Linking 解析)
**作为** 一个接收者（如开发人员）
**我希望** 在浏览器中打开带参数的分享链接时，系统能直接展示该文档
**以便于** 我不需要再手动去文件树里寻找。

**验收标准 (Acceptance Criteria):**
1. 系统首页加载时，优先解析 URL 中的 `repo` 和 `file` 参数。
2. 若存在参数，系统必须自动在左侧文件树中找到对应的仓库，并逐层展开文件夹，直到目标节点，并给予高亮显示。
3. 如果是文件节点，右侧主阅读区自动发起内容请求并渲染该文件。
4. 在文件树异步加载数据的过程中，右侧阅读区应有合适的 Loading 骨架屏状态。

## Story 4: 优雅的 404 异常处理
**作为** 一个接收者
**我希望** 当我打开一个无效的分享链接时，系统能明确告诉我原因
**以便于** 我知道是文件不存在而不是系统坏了。

**验收标准 (Acceptance Criteria):**
1. 如果 URL 中的 `repo` 不在当前系统的白名单配置中，右侧阅读区展示 404/403 缺省页（提示：仓库无权限或未配置）。
2. 如果 URL 指向的 `file` 在仓库中不存在（已被删除或重命名），右侧阅读区展示 404 缺省页（提示：文件不存在或已被删除）。
3. 缺省页设计应包含符合产品调性的插画/Icon，以及清晰的文案说明。
4. 发生异常时，左侧文件树应保持正常加载（只加载可见的白名单仓库），不应整个页面崩溃。
