# 概念设计 (Concept Design)

## 需求背景
为解决用户在交互式 PRD (`Interactive_PRD.html`) 中无法全屏查看右侧原型页面的痛点，需要在右侧 iframe 容器的右上角增加一个常驻半透明的悬浮图标。点击该图标时，需要在新的浏览器标签页中打开当前 iframe 对应的原型 HTML。

## 关键技术点：URL 获取机制的方案推演

在点击“新标签打开”按钮时，如何获取当前正确的原型 URL？这里存在两种主要的概念方案：

### 方案 A：直接读取 iframe DOM 的 src 属性
即在按钮的点击事件中，通过 JavaScript 执行 `const url = document.getElementById('prototype-frame').src; window.open(url, '_blank');`。

*   **优点 (Pros)**：
    1. **实现最简单直接**：仅需要一行代码，无需维护复杂的外部状态变量。
    2. **所见即所得**：无论 iframe 内部是如何加载的，它的 `src` 就是当前浏览器认为的源地址。
*   **缺点 (Cons)**：
    1. **状态丢失风险**：如果原型内部是一个单页应用 (SPA)，或者仅仅是通过内部 Hash 跳转（如从 `index.html#section-1` 跳到 `index.html#section-2`），由于同源策略或浏览器的历史记录机制，外层 iframe 元素的 `src` 属性值可能**不会实时更新**（依然保留着初始加载时的 URL）。
    2. **跨域限制**：在某些本地文件协议 (`file://`) 下，浏览器对 iframe 的跨文档通信有严格限制，直接读取 `src` 可能在特定安全策略下失效或报错。

### 方案 B：读取外层页面维护的全局状态 / URL Hash
交互式 PRD 本身就依赖外层 URL Hash (如 `http://localhost/Interactive_PRD.html#F1_Home/index.html`) 来驱动左侧目录高亮和右侧 iframe 加载。在此方案中，按钮点击时不读 iframe，而是读取外层维护的 active 状态或直接解析外层页面的 `window.location.hash`。

*   **优点 (Pros)**：
    1. **状态绝对同步**：由于左侧 TOC 和右侧 iframe 的展示是由外层全局状态统一管理的，读取该状态能确保“新标签页打开”的永远是逻辑上当前激活的页面，携带正确的锚点，不会丢失上下文。
    2. **安全可靠**：完全避开了 iframe 跨域或文件协议下的安全限制，因为所有状态都是在外层主文档中维护的。
*   **缺点 (Cons)**：
    1. **实现略微复杂**：需要解析当前的外层 hash（例如截取 `#` 后面的路径 `F1_Home/index.html`），再拼接成新窗口的完整 URL（例如 `./F1_Home/index.html`）。
    2. **如果 iframe 内部发生不受外层监控的跳转**，外层状态可能会与实际显示的页面不一致。但这在交互式 PRD 场景下几乎不发生，因为导航主要由左侧目录驱动。

## 结论与推荐
综上所述，考虑到交互式 PRD 的核心是基于 Hash 联动，且需要保证状态的精确同步，**强烈推荐采用 方案 B (读取外层全局状态 / URL Hash)**。这不仅能避免状态丢失，还能规避本地查看时的跨域报错风险。

---
*注：视觉方案已确认采用“常驻半透明”策略，以防遮挡原型自带的交互组件。*
