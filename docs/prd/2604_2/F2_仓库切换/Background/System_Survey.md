# 系统调研与上下文分析 (System_Survey) - F2_仓库切换

## 1. 调研目标
分析在现有 PRD-Reader 系统中剥离“独立仓库列表页”并在阅读器中增加“下拉仓库切换”功能对现有架构、路由及数据流转的影响，确保新设计与原有系统逻辑兼容。

## 2. 现有系统核心逻辑分析

### 2.1 路由与入口逻辑 (Routing & Entry)
*   **原有逻辑**：用户访问根路由 `/`，系统判断若已配置（Token 与白名单存在），则直接渲染 `Reader` 页面，并拉取**所有**白名单仓库的文件树，将它们混合展示在左侧导航栏。
*   **调研结论 (冲突点)**：原有逻辑将所有仓库混杂展示，无法满足“按仓库隔离阅读”的需求。
*   **设计约束 (应对方案)**：
    *   **首页变更**：根路由 `/` 的行为变更为渲染“独立仓库列表页”（前提是已配置且无具体参数）。
    *   **Reader 路由**：必须依赖 `?repo=` 参数。只有当 URL 中携带明确的 `?repo=` 时，才渲染 `Reader` 页面并仅拉取该仓库的文件树。
    *   **历史链接处理**：若用户访问旧的分享链接（仅有 `?file=` 无 `?repo=`），系统将直接重定向至根路由 `/` 的“仓库列表页”，要求用户重新选择入口，不再尝试全库搜索，以降低性能消耗和逻辑复杂度。

### 2.2 数据来源与权限 (Data Source)
*   **原有逻辑**：后台 `Admin` 页面负责配置 GitHub/GitLab 的 Token 以及“白名单仓库列表”（以字符串形式存储在 config 中）。
*   **调研结论**：新需求需要展示一个仓库列表。如果调用平台 API 动态获取用户的全量仓库，可能会引发隐私和权限安全问题，且与当前“只暴露白名单内容”的安全策略相悖。
*   **设计约束 (应对方案)**：直接复用系统现有的 `Config` 中存储的“白名单仓库列表”作为独立列表页的数据源。仅向用户展示管理员允许公开的仓库。

### 2.3 Reader 视图状态流转 (State Machine)
*   **原有逻辑**：`Reader` 页面加载时触发全局的 `fetchTree`，随后将数据喂给左侧的文件树组件。
*   **调研结论**：新增左上角的“下拉仓库切换”会触发频繁的数据重载。
*   **设计约束 (应对方案)**：在 `Reader` 页面的 Header 左上角植入一个受控的 Dropdown 组件。当选择变更时，必须：
    1.  更新 URL 中的 `?repo=` 参数（通过 history API 或 router 跳转）。
    2.  清空当前的 `?file=` 参数（因为文件路径已变更）。
    3.  重置 `ReaderStore` 中的当前文件内容，并触发针对新 `repo` 的 `fetchTree`。
    4.  展示 Loading 状态直至新文件树渲染完毕。

## 3. 受影响的现有页面/模块
1.  `Home.tsx` / 路由入口模块：需要增加判断逻辑，拦截无 `?repo=` 的访问并重定向/渲染仓库列表。
2.  `Reader.tsx` (布局与 Header)：左上角原有显示标题的位置需重构为可点击交互的 Dropdown 组件，并增加返回列表页的入口。
3.  `readerStore.ts` / 状态管理：`fetchTree` 等核心 Action 的入参需强化对单一 `repo` 的支持（虽然底层 API 已支持，但前端调用层需严格隔离）。

## 4. 约束与依赖
*   **依赖**：完全依赖现有后台配置的白名单数组。若白名单为空，新页面需展示友好的引导提示（引导 Admin 配置）。
*   **向下兼容约束**：废弃所有无 `?repo=` 的旧文档链接，统一进行重定向降级处理，不提供后向查找兼容。
