# 概念设计与头脑风暴 (Concept Design)

## 1. 需求回顾
针对现有的 PRD-Reader 侧边栏，优化长文件名的显示效果。已确定的核心交互包括：增加基础宽度、缩小字号、拖拽调整宽度、尾部单行截断（纯 CSS）以及 Hover 显示自定义 Tooltip。

## 2. 概念方案推演
在具体实现“截断 + Tooltip”的交互上，存在以下两种体验维度的设计方向：

### 方案 A：全局强制 Tooltip (Global Tooltip)
*   **概念说明**：无论文本是否真的发生了截断，只要用户的鼠标悬停在**任何**一个文件或文件夹节点上，都统一弹出 Tooltip 显示完整名称。
*   **Pros (优点)**：
    *   **实现简单**：不需要动态计算文本的实际渲染宽度和容器宽度的差值。
    *   **行为一致**：用户的悬停预期是确定的，只要悬停就有提示。
*   **Cons (缺点)**：
    *   **信息冗余**：对于那些完全显示在视野内（没有被截断）的短文件名，弹出 Tooltip 是多余的，反而会遮挡下方的文件，造成视觉干扰（“牛皮癣”效应）。
    *   **性能浪费**：频繁触发的 Tooltip 渲染。

### 方案 B：智能按需 Tooltip (Smart Tooltip / On-Demand)
*   **概念说明**：只有当文件名的实际长度**超出了**当前侧边栏分配给该节点的显示宽度，并**真正发生了**省略号截断时，鼠标悬停才会触发 Tooltip。未截断的短文件名悬停时没有任何反应。
*   **Pros (优点)**：
    *   **极致体验 (克制)**：符合“Don't make me think”的设计原则，只在用户需要的时候提供帮助，保持界面清爽。
    *   **降低干扰**：有效避免了大量短文件名的无效 Tooltip 遮挡问题。
*   **Cons (缺点)**：
    *   **实现复杂度增加**：前端需要实现一套逻辑（通常是通过比较 DOM 元素的 `scrollWidth` 和 `clientWidth`）来判断当前元素是否溢出。

## 3. 方案决策
*   **最终选择**：**方案 B：智能按需 Tooltip (Smart Tooltip / On-Demand)**
*   **决策原因**：
    *   PRD-Reader 作为一款阅读类工具，核心诉求是提供**沉浸、无干扰**的阅读体验。全局强制 Tooltip（方案 A）会带来严重的视觉干扰，与阅读工具的调性相悖。
    *   虽然方案 B 增加了部分实现复杂度，但目前通过 React 的 Hooks 或原生的 ResizeObserver/offsetWidth 计算，实现“判断是否溢出”的技术已经非常成熟，性能开销可控。
    *   结合“拖拽调整侧边栏宽度”的功能，方案 B 能做到真正的智能：当用户拉宽侧边栏使得长文件名完全显示后，原本出现的 Tooltip 会智能消失，体验更加流畅自然。
